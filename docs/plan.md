# Habit Flywheel vNext Plan

## 目标

将当前项目从“带奖励机制的习惯管理工具”升级为“以奖励验证为核心的个人行动系统”。

vNext 的默认视角不再是管理模块，而是管理正在追求的奖励计划。用户进入应用后，应该先看到“我正在为哪些奖励行动、今天最值得做什么、哪些欲望已经被验证或淘汰”。

## 已确认的产品方向

- 定位：公开 Beta 产品，但核心服务深度自驱用户。
- 能量模型：混合模式。
  - 奖励专属能量用于推进具体奖励。
  - 全局累计能量用于长期成就、历史统计和成长度量。
- 奖励类型：
  - 一次性大目标
  - 可重复小奖励
- 习惯机制：支持 `daily / weekly / monthly` 目标制，不再停留在表单占位。
- 绑定关系：不再作为一级导航独立存在，改为奖励详情中的内嵌能力。
- 视觉方向：采用更克制的“纸白 / 墨绿 / 铜色”体系，强调“欲望验证”和“行动兑现”的仪式感，而不是延续紫色默认 AI 风。
- 视觉继承：保留深浅色主题、移动端优先和 PWA 基础能力；旧版紫金识别只保留为历史参考，不再作为默认主色。
- 视觉分层：弹窗层作为高意图操作层单独设计，不与普通卡片共用同一套强调方式。

## 当前版本诊断

### 优势

- 核心理念清晰，欢迎页已经在传达“先奖励，再验证，再行动”的产品思想。
- 习惯、奖励、完成记录、用户能量的基础数据结构已具备。
- 已有多语言、PWA、认证、历史统计等正式产品需要的基础设施。
- 现有打卡、取消打卡、乐观更新、错误回滚等交互语义已经形成可复用基础，不应在重构中丢失。

### 主要问题

1. 应用主流程仍是开发者视角。
   - 当前导航围绕 `今日 / 习惯 / 奖励 / 绑定 / 历史 / 设置` 展开。
   - 用户首先看到的是模块集合，而不是当前目标。
2. 产品叙事与应用内体验脱节。
   - 欢迎页强调“奖励验证”。
   - 应用内却仍以习惯 CRUD 为中心。
3. 数据模型没有完整承接产品承诺。
   - 习惯频率、目标次数已出现在表单中，但未真正落到数据库和业务逻辑。
4. 若干能力仍停留在壳层。
   - 设置中的导入、导出、清空、重置暂无实装。
   - 多语言未完全覆盖。
5. 工程完成度未收口。
   - `lint` 未通过。
   - 首页和应用体量偏大，存在后续拆包空间。

## 产品原则

1. Reward-first
   - 奖励是一级对象，习惯是奖励计划的执行路径。
2. Validation over possession
   - 奖励不是购物清单，而是待验证的欲望。
3. Execution over management
   - 首页优先回答“今天做什么最值”，而不是“可以管理哪些模块”。
4. Feedback with consequence
   - 每次打卡都要明确推进某个目标，而不是仅显示抽象积分。
5. Review for self-knowledge
   - 历史页不只统计次数，要帮助用户识别“值得投入的欲望”和“被过滤掉的冲动”。
6. Visual hierarchy with consequence
   - 页面层、卡片层、弹窗层必须有清晰视觉分层；兑现、放弃、创建等高语义操作不能只靠同一套中性卡片样式承载。

## 视觉系统补充

详细方案见 [docs/visual-system.md](./visual-system.md)。

### 全局方向

- 主背景改为暖纸白 / 浅矿物灰，不再默认纯白 + 紫色发光。
- 主色由高饱和紫色改为深青绿 / 墨绿，负责“执行、推进、可信赖”。
- 强调色改为铜色 / 琥珀棕，负责“奖励、阈值、兑现、里程碑”。
- 危险动作改为低饱和铁锈红，避免亮红带来的工具警报感。

### 状态色语义

- `wish`：中性灰
- `validating`：铜棕 / 沙金
- `active`：深青绿
- `ready_to_redeem`：亮铜色
- `redeemed`：苔绿 / 深绿色
- `abandoned`：灰红 / 铁锈红

### 弹窗层规则

弹窗是高意图操作层，不能只复用普通页面卡片样式。

- `PlanComposer`
  - 中性底 + 深青绿主按钮 + 轻铜色步骤强调
- `PlanDetail`
  - 中性运营面板，状态色只用于局部标签和操作
- `RedeemPlanDialog`
  - 铜色强化“兑现仪式感”，但背景保持克制
- `AbandonPlanDialog`
  - 使用安静的灰红 / 铁锈红语义，不做强刺激红黑警报风
- `LegacyWorkspaceDialog`
  - 去饱和、偏灰，明确它是兼容层而不是主舞台

### 欢迎页方向

- 欢迎页改为更强的编辑感宣言页，而不是紫色产品落地页。
- Hero 区不再使用顶部紫色 radial glow，改为暖纸白 + 极浅青灰 / 铜色块面。
- 预览卡片应像“产品样本页”，而不是发光 UI demo。
- CTA 使用深青绿主按钮和铜色 / 中性色次按钮。

## 保留与迁移约束

1. 不做推倒重来式重写
   - 采用分阶段迁移，让现有可运行能力持续可用。
2. 保留现有基础设施
   - 继续沿用现有认证、PWA、国际化、深浅色主题和响应式框架。
3. 保留现有关键交互语义
   - 打卡、取消打卡、乐观更新、失败回滚、能量同步等行为语义作为 vNext 的稳定基础。
4. 保留品牌连续性
   - 紫金色与能量符号体系可继续沿用，但页面结构和视觉层级应从“模块管理”转向“目标推进”。
5. 优先兼容当前路由与部署方式
   - `Welcome / Auth / App` 的整体入口结构可延续，主改造集中在认证后的应用壳层。

## vNext 信息架构

### 一级导航

- Dashboard
  - 飞轮控制台，展示当前最重要的奖励计划、今日行动和进展状态。
- Plans
  - 所有奖励计划的列表与分组视图。
- Review
  - 周/月复盘、投入分布、验证结果、能量趋势。
- Library
  - 通用习惯模板、归档奖励、已完成计划、被放弃计划。
- Settings
  - 账户、语言、通知、数据管理、实验功能。

说明：
- 现有 `Bindings` 页面被移除。
- `Habits` 与 `Rewards` 不再平级成为主导航；它们被收编进 `Plans` 和计划详情。

## 核心对象模型

### Reward Plan

奖励从单一记录升级为计划对象，建议包含：

- `name`
- `description`
- `type`
  - `one_time`
  - `repeatable`
- `status`
  - `wish`
  - `validating`
  - `active`
  - `ready_to_redeem`
  - `redeemed`
  - `abandoned`
- `energy_cost`
- `current_energy`
- `target_date`
- `repeat_rule`（可重复奖励需要）
- `priority`
- `motivation_note`

### Habit

习惯从“简单打卡项”升级为行动规则，建议包含：

- `name`
- `description`
- `frequency`
  - `daily`
  - `weekly`
  - `monthly`
- `target_count`
- `energy_value`
- `is_archived`
- `default_plan_id` 或通过单独绑定表支持多计划关系

### Plan Binding

若延续“一个奖励绑定多个习惯”的设计，建议显式建表，而不是只在习惯表上放一个外键：

- `plan_id`
- `habit_id`
- `allocation_mode`
  - `exclusive`
  - `shared`
- `allocation_ratio`

说明：
- 若 vNext 第一阶段想控制复杂度，可以先实现 `exclusive` 绑定。
- 现有的“一个习惯绑定一个奖励”仍可保留为初版约束，但数据结构应为后续扩展预留空间。

### Completion

完成记录建议增加：

- `completed_at`
- `energy_gained`
- `note`
- `evidence_type`（可选，未来支持图片/链接）
- `plan_id_snapshot`
- `frequency_bucket`

### User Energy

保留 `user_energy.total_energy`，作为全局累计能量，不直接替代奖励专属进度。

## 核心页面设计

### 1. Dashboard

目标：让用户打开应用后立即知道今天最值得做什么。

建议模块：

- Hero 区
  - 当前主计划
  - 距离兑现还差多少
  - 今天完成后能推进多少
- Today Actions
  - 今日应做习惯
  - 按“影响最大”排序，而不是按创建时间排序
- Momentum Strip
  - 连续行动天数
  - 本周计划推进度
  - 累计全局能量
- Validation Signals
  - 最近完成的计划
  - 最近放弃的欲望
  - 最近最投入的领域

视觉要求：

- Hero 与里程碑不再采用大面积紫金渐变。
- 当前主计划和执行层用深青绿建立秩序。
- 奖励阈值、兑现机会和里程碑用铜色点亮。

### 2. Plans

目标：管理所有奖励计划，而不是零散管理奖励和绑定关系。

建议视图：

- `Active`
- `Validating`
- `Ready`
- `Redeemed`
- `Abandoned`

卡片重点信息：

- 奖励名称
- 当前状态
- 当前能量 / 所需能量
- 绑定习惯数量
- 预计完成时间
- 最近推进时间

### 3. Plan Detail

这是 vNext 的核心页面。

建议模块：

- 计划概览
- 动机说明
- 绑定习惯列表
- 自动计算的推进预估
- 近期完成记录
- 状态操作
  - 进入验证
  - 激活计划
  - 标记可兑现
  - 兑现
  - 放弃

绑定管理在这里完成，不再单独给一个 `Bindings` 页面。

视觉要求：

- 计划详情是“运营面板”，不是宣传卡片。
- 状态标签、危险操作、兑现入口要有明确颜色分工。
- 弹窗内的局部高亮要服务于状态判断，而不是制造泛用炫光效果。

### 4. Quick Capture / Create Flow

首次创建流程应从奖励开始：

1. 我想要什么奖励
2. 为什么想要它
3. 它属于一次性还是可重复奖励
4. 我要用哪些习惯来验证它
5. 每种习惯完成一次推进多少
6. 什么时候算值得兑现

新用户不应先建习惯再思考奖励。

### 5. Review

复盘页要回答四个问题：

1. 我最近真正把时间和行动投给了什么
2. 哪些奖励正在被稳定推进
3. 哪些奖励其实并不值得我继续追
4. 哪些习惯最有效地推动了目标

建议模块：

- 周/月能量趋势
- 奖励推进排行
- 习惯贡献排行
- 已验证成功的奖励
- 被过滤掉的欲望
- 行动密度热力图

### 6. Welcome

目标：在未登录状态下先传达“欲望验证”方法论，而不是展示一个紫色 SaaS 首页。

建议方向：

- 改成编辑感宣言页
- 使用纸白 / 墨绿 / 铜色体系
- 预览卡片更像产品样本页而不是发光 Demo
- 功能介绍区通过中性分层和少量强调色建立秩序

## 关键交互设计

### 打卡反馈

当前打卡反馈主要是“完成了 + 获得能量”。

vNext 要升级为：

- 本次完成推进了哪个计划
- 推进了多少
- 距离兑现还差多少
- 如果达成关键阈值，触发更强的反馈文案与动画

### 兑现反馈

兑现不应只是按钮状态变化，需要最小化仪式感设计：

- 兑现确认层
- 本次奖励由哪些习惯共同换来
- 花了多久完成
- 总共执行了多少次行动
- 兑现后是否生成下一阶段计划

视觉要求：

- 兑现确认层应在弹窗中形成“仪式感”而不是“领奖页”。
- 只强化关键数字、确认按钮和少量铜色描边，不做整页高饱和铺色。

### 放弃反馈

“放弃”不是失败，而是验证完成。

建议明确记录：

- 放弃原因
- 已投入多少行动
- 学到什么

视觉要求：

- 放弃弹窗应克制、安静、偏反思语义。
- 使用低饱和灰红，而不是泛用亮红警报色。

这会成为产品理念的重要组成部分。

## 数据与工程实施计划

### Phase 1：产品骨架重构

目标：先把主流程改对。

- 重构一级导航与首页结构
- 用 `Dashboard + Plans + Review + Settings` 取代当前模块式布局
- 将 `Bindings` 吸收到计划详情
- 新增“从奖励开始”的创建流程
- 尽量复用现有 `/`、`/auth`、`/app` 路由结构，避免大范围入口层改造
- 保持现有 Supabase 数据可兼容运行

### Phase 2：数据模型补全

目标：让业务逻辑真正支持产品承诺。

- 为习惯补齐 `frequency` 与 `target_count`
- 为奖励补齐 `type`、`status`、`target_date`、`motivation_note`
- 引入更清晰的绑定关系模型
- 为完成记录增加备注与计划快照

### Phase 3：复盘与洞察

目标：把“飞轮”做成可感知系统。

- 重构历史页为 Review
- 增加计划推进、欲望验证、习惯贡献等分析维度
- 增加关键阈值反馈和兑现仪式

### Phase 4：工程收口

目标：从“能跑”变为“可持续维护”。

- 补齐 i18n 覆盖
- 解决 `lint` 错误
- 为首页与复盘页做按路由拆包
- 清理未实现设置项或补完实现
- 统一类型定义，去除主要 `any`
- 增加基础测试、性能检查和关键交互回归验证

### Phase 5：视觉系统重构

目标：让欢迎页、认证后主壳、状态卡片和弹窗层共享统一视觉语义。

- 在 `src/index.css` 中重构全局 light / dark token
- 重构认证后壳层、Dashboard 和核心卡片的配色结构
- 单独收口 `PlanComposer / PlanDetail / Redeem / Abandon / LegacyWorkspace` 弹窗层
- 重做欢迎页 Hero、预览卡片和 CTA 的配色体系
- 为计划状态建立一眼可辨的颜色语义

## 优先级

### P0

- 重构首页为飞轮控制台
- 重构奖励为计划视角
- 将绑定管理收编进计划详情
- 打通从奖励开始的创建流程

### P1

- 数据模型升级
- Review 复盘页重构
- 兑现与放弃流程设计

### P2

- 导入导出
- 成就系统
- 证据上传
- 模板库与推荐计划
- PWA 体验增强与通知能力补完
- 视觉系统重构

## 成功指标

### 产品指标

- 新用户首次创建流程中，先创建奖励计划的比例显著高于先创建习惯。
- 活跃用户中，至少有一个 `active` 或 `validating` 计划的比例持续提升。
- 奖励兑换记录中，来自绑定习惯稳定推进的比例提升。
- 被放弃计划也能被记录和回看，体现“验证”价值，而不是只保留成功案例。

### 体验指标

- 用户进入首页 5 秒内能理解当前主目标。
- 打卡后 1 秒内能看见目标推进反馈。
- 主要页面均支持中英日完整文案。
- `lint` 与 `build` 均通过。

## 结论

vNext 的核心不是继续叠加模块，而是把“欲望验证”真正做成应用的主操作系统。

只要首页、计划模型、创建流程和复盘逻辑围绕这个核心重新组织，这个项目就会从一个有想法的工具，升级成一个真正有方法论和辨识度的产品。
